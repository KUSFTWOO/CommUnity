# [2026-05-29] 게시글 검색 기능 구현

## 담당자
- AI (Claude Haiku 4.5)

## 소요 시간
- 예상: 1시간
- 실제: 45분

## 구현 기능

### 완성된 주요 기능
1. **검색 페이지** (`/search/`)
   - 검색창과 "검색" 버튼만 있는 깔끔한 UI
   - 초기 상태: 안내 메시지 표시
   - 검색 후: 결과 목록 표시

2. **검색 로직**
   - 공지사항(Notice) 검색: title과 content에서 키워드 검색
   - 게시판(Post) 검색: title과 content에서 키워드 검색
   - 대소문자 구분 없음 (icontains 사용)
   - Q 객체로 OR 쿼리 조합

3. **검색 결과 표시**
   - 결과 요약: "키워드" 검색 결과 X건
   - 각 결과 카드:
     - 타입 배지 (📢 공지사항 / 💬 게시판)
     - 제목 (링크)
     - 본문 요약 (최대 100자)
     - 작성자
     - 작성일
   - 최신순 정렬

4. **사용자 경험**
   - navbar에 🔍 검색 링크 추가
   - 검색 결과 없음 메시지
   - 결과 카드 호버 효과

## 기술 구현 사항

### Views
- `search_view(request)`: GET 파라미터 'q'에서 키워드 받음
- Notice와 Post 객체를 통합하여 결과 생성
- created_at 기준으로 최신순 정렬

### URLs
- `/search/?q=키워드` → 검색 결과 페이지

### Templates
- `search/search.html`
  - 검색 폼 (GET 방식)
  - 결과 요약 (Tailwind 배지)
  - 결과 카드 (타입별 색상 구분)
  - line-clamp CSS로 본문 요약 2줄 제한

### 통합
- settings.py: INSTALLED_APPS에 "apps.search" 추가
- config/urls.py: "search/" URL 등록
- navbar.html: 검색 링크 추가

## 설계 결정사항

### 왜 search 앱을 별도로 만들었나?
- 검색은 여러 모델(Notice, Post)을 다루므로, 단일 책임 원칙에 맞게 별도 앱으로 분리
- 향후 확장성 고려 (파일 검색, 댓글 검색 등)

### 왜 GET 방식을 사용했나?
- URL에 검색 쿼리가 남아 북마크 가능
- 브라우저 뒤로가기로 이전 검색 복원
- 공유 가능한 링크 생성

### 왜 icontains를 사용했나?
- 대소문자 구분 없이 검색 (UX 개선)
- 정확한 검색도 포함
